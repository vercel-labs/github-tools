`@github-tools/sdk` turns the [GitHub REST API](https://docs.github.com/en/rest) into **42 typed, AI-callable tools** your agent can use out of the box — repositories, pull requests, issues, commits, code search, gists, and GitHub Actions.

Instead of hand-writing Zod schemas, Octokit calls, and result shaping for every endpoint, you get production-ready tool definitions with **presets** to scope capabilities, **human approval** built into every write operation, and **commit attribution** for bot identity — ready to drop into `generateText`, an agent loop, or a durable workflow.

## Why an SDK for GitHub tools?

Wiring GitHub into an AI agent by hand means writing a tool definition per endpoint: an input schema the model can fill reliably, the Octokit call, output shaping so raw API payloads don't blow up your context window, and a safety gate so the model can't merge a PR without asking. Multiply that by every operation your agent needs.

The SDK does all of it:

- **Typed inputs** — every tool ships a [Zod](https://zod.dev) schema with described fields, so models fill arguments correctly.
- **Shaped outputs** — responses are trimmed to what the model needs, not raw REST payloads.
- **Write safety by default** — all 20 write tools require [human approval](/guide/approval-control) unless you opt out.
- **Scoped capability** — [presets](/guide/presets) expose only the tools a workflow needs.
- **Durable by design** — every tool runs as a `"use step"` boundary, so calls are retryable workflow steps under [Vercel Workflow](/frameworks/vercel-workflow).

## A GitHub agent in 3 files

The fastest path from zero to a working GitHub agent is [eve](https://eve.dev), Vercel's filesystem-first agent framework. One tools file registers all 42 tools:

::code-group
```md [agent/instructions.md]
You are a GitHub code-review assistant.
```

```ts [agent/agent.ts]
import { defineAgent } from 'eve'

export default defineAgent({
  model: 'anthropic/claude-sonnet-5',
})
```

```ts [agent/tools/github.ts]
import { createGithubTools } from '@github-tools/sdk/eve'

export default createGithubTools({
  preset: 'maintainer',
})
```
::

Run `npx eve dev` and you have a GitHub agent with durable human-in-the-loop approval on every write. Full walkthrough: [Build a GitHub agent with eve](/frameworks/eve).

## Works with your framework

The same tools plug into every major agent runtime — pick the entry point that matches your stack:

| Framework | Import | Best for |
|---|---|---|
| [AI SDK](/frameworks/ai-sdk) | `@github-tools/sdk` | `generateText`, `streamText`, `ToolLoopAgent` — scripts, chat backends, existing AI SDK apps |
| [eve](/frameworks/eve) | `@github-tools/sdk/eve` | Filesystem-first agents in 3 files, durable approval (`once`, predicates) |
| [Vercel Workflow](/frameworks/vercel-workflow) | `@github-tools/sdk/workflow` | Crash-safe, retryable agents that survive restarts and timeouts |
| [Chat SDK](/frameworks/chat-sdk) | `@github-tools/sdk` + `chat` | GitHub, Slack, and Discord bots with durable multi-turn sessions |

## Explore the tools

The SDK ships **42 tools** covering repositories, branches, pull requests, issues, commits, code search, gists, and workflows. Each tool wraps a GitHub API operation (mostly REST; line-level blame uses the GraphQL `Commit.blame` field) and is fully typed with [Zod](https://zod.dev) schemas.

Browse the full list in the [Tools Catalog](/api/tools-catalog).

## Compose tools with presets

Instead of wiring tools individually, use a **preset** to get a curated subset for a specific workflow:

| Preset | Tools included | Use case |
|---|---|---|
| `repo-explorer` | repository metadata, branches, file content, code search, gists, workflows | knowledge retrieval, repo Q&A |
| `code-review` | pull requests, commits, file diffs, review comments | PR copilots, change summaries |
| `issue-triage` | issues, labels, comments, close/create | support triage, backlog bots |
| `ci-ops` | workflows, runs, jobs, commits, repository context | CI monitoring, build ops |
| `maintainer` | all tool families | full operator workflows |

Presets compose as arrays: `preset: ['code-review', 'issue-triage']`. Learn more in [Scope with Presets](/guide/presets).

## Safety is not an afterthought

Every write operation — merging a PR, pushing a file, closing an issue — is gated behind approval **by default**. You choose the policy:

```ts
createGithubTools({
  requireApproval: {
    mergePullRequest: true,   // always ask
    addIssueComment: false,   // never ask
  },
})
```

On eve, approval goes further with `'once'` (approve once per session) and input-dependent predicates. Combine with least-privilege [token scopes](/guide/tokens-and-auth) for defense in depth. Details: [Control Write Safety](/guide/approval-control).
